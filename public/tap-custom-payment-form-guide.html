💡 ليش الـ Card SDK مهم؟

✗ الفلو الحالي (Hosted)

العميل يضغط "ادفع" → يتحوّل لصفحة Tap الخارجية → يرجع لموقعك.

  • conversion rate أقل بسبب الـ redirect
  • المستخدم يحس إنه طلع برّا الموقع

✓ الفلو المستهدف (Embedded SDK)

حقول البطاقة تظهر داخل صفحة الـ checkout عندك كـ iframes آمنة من Tap.

  • تجربة أسلس → conversion أعلى
  • PCI compliance يبقى على Tap (مش عليك)
  • 3DS فقط يفتح iframe صغير عند الحاجة

🔧 الوضع الحالي عندك في الكود

راجعت الـ tap-create-charge Edge Function عندك. النتيجة:

// supabase/functions/tap-create-charge/index.ts
source: {
  id: token_id || "src_all"   // ← الـ backend جاهز يستقبل token!
}
خبر طيّب: الـ backend عندك بالفعل يقبل token_id اللي بيرجع من الـ SDK. الناقص فقط هو دمج الـ SDK في الـ frontend وتمرير الـ token بدل src_all. يعني التعديل أصغر مما تتوقع.

1 Card SDK Web v2

الرابط الرسمي: developers.tap.company/docs/card-sdk-web-v2

كيف يشتغل؟

  1. تضيف script للـ SDK في الصفحة.
  2. تستدعي tap.create({ publicKey }) ثم card.mount(...) — تظهر الحقول كـ iframes.
  3. عند الضغط على "ادفع" تستدعي tap.tokenize() → ترجعلك tok_xxxxx.
  4. تبعت الـ tok_xxxxx للـ Edge Function tap-create-charge.
  5. الـ backend يكمّل بنفس الكود الموجود حالياً.

سكتش بسيط للـ frontend

// CheckoutPaymentStep.tsx
const tap = window.Tap(publicKey);
const elements = tap.elements();
const card = elements.create("card");
card.mount("#card-element");

async function handlePay() {
  const { id, error } = await tap.tokenize(card);
  if (error) return showError(error.message);

  // نفس الـ Edge Function — بس مع token_id
  await supabase.functions.invoke("tap-create-charge", {
    body: { course_id, amount, token_id: id }
  });
}

2 مفاتيح API — من TapOS

  1. ادخل os.tap.company
  2. اذهب إلى Accounts → Operators → (Merchant)
  3. تجد ٤ مفاتيح:
المفتاحوين يستخدم؟
pk_test_xxxfrontend SDK · بيئة الـ staging
sk_test_xxxbackend · بيئة الـ staging
pk_live_xxxfrontend SDK · production
sk_live_xxxbackend · production
تحذير: الـ sk_* ما يلمس الـ frontend أبداً — يبقى في Supabase Secrets فقط (موجود الآن باسم TAP_SECRET_KEY). الـ pk_* هو اللي يتحط في الـ frontend.

3 Apple Pay (اختياري — لكن يرفع الـ conversion كثير)

  1. Tap بترسلك ملف apple-developer-merchantid-domain-association.
  2. تحطه في: 
    public/.well-known/apple-developer-merchantid-domain-association
    ليصبح متاح على: 
    https://academy.bikerz.com/.well-known/apple-developer-merchantid-domain-association
  3. ترد على Rawan تأكد إن الملف up.
  4. Tap تعمل whitelist للدومين + تفعّل Apple Pay على حسابك.

متطلبات Apple Pay

  • ✓ HTTPS — موجود عندك
  • ✓ Public key في الفرونت
  • ✗ Localhost ما يشتغل — لازم بيئة فعلية
  • ✓ Apple Pay مفعّل على حساب Tap

دليل الدمج: developers.tap.company/docs/apple-pay-web

📋 خطة التنفيذ المقترحة

1 استخراج المفاتيح وإعداد البيئة

  • جيب pk_test و sk_test من TapOS.
  • ضيف VITE_TAP_PUBLIC_KEY في .env.local.
  • تأكد إن TAP_SECRET_KEY محدّث في Supabase secrets.

2 دمج الـ SDK في checkout

  • ضيف script tag للـ SDK في index.html.
  • عدّل CheckoutPaymentStep.tsx ليعرض حقول Tap بدل redirect.
  • عند submit: استدعي tap.tokenize() ومرّر الـ token_id للـ Edge Function.

3 اختبار

  • بكروت Tap التجريبية (مفصّلة في توثيقهم).
  • تأكد من 3DS callback — لازم يرجّعك للموقع تلقائياً.

4 Apple Pay (Phase 2)

  • ارفع ملف .well-known/.
  • أكد لـ Rawan ليتم whitelist.
  • ضيف زر Apple Pay فوق الفورم العادي.
